## KMeans Gap/Stability Benchmark: Parallelism And Outputs

This note documents the parallel execution settings for
`msd_deezer_workspace/kmeans_k_selection_benchmark/`.

The benchmark does use parallel processing. The worker/thread/process count is
now set to 16 everywhere that has an explicit benchmark-level worker setting.

## Parallel Stages

| Stage | File | Parallel mechanism | Worker setting |
|---|---|---|---|
| 10k sample materialisation | `select_random_sample.py` | `ThreadPoolExecutor` for copy/symlink I/O | `--workers`, default `16` |
| Dual preprocessing | `preprocess_downloaded_audio.py` -> `audio_preprocessing/processor.py` | `ProcessPoolExecutor` over audio files | `DEFAULT_WORKERS = 16` |
| Handcrafted feature extraction | `extract_audio_features.py` | `ProcessPoolExecutor` over audio files | run script passes `--workers 16` |
| Pretrained MusicNN extraction | `run_parallel_extraction.sh` | 16 independent CPU shard processes | `run_parallel_extraction.sh 16` |
| Pretrained MERT extraction | `run_parallel_extraction.sh` | 1 GPU process, concurrent with EnCodecMAE and MusicNN shards | GPU prefetch threads default `16` |
| Pretrained EnCodecMAE extraction | `run_parallel_extraction.sh` | 1 GPU process, concurrent with MERT and MusicNN shards | GPU prefetch threads default `16` |
| Sharded embedding merge | `merge_sharded_embeddings.py` | `ThreadPoolExecutor` over song stems | `--workers`, default `16` |
| KMeans candidate evaluation | `run_kmeans_gap_stability.py` | `joblib.Parallel` over candidate K values | `--workers`, default `16`; orchestrator passes `--workers 16` |
| Gap-statistic reference fits | `run_kmeans_gap_stability.py` | `joblib.Parallel` over `(K, reference sample)` jobs | same `--workers 16` |
| Prediction-strength fits | `run_kmeans_gap_stability.py` | `joblib.Parallel` over candidate K values | same `--workers 16` |

## Serial Stages

The orchestrator does not launch the six final K-selection runs at the same
time. It runs them one by one:

- `features` x `pca_only`
- `features` x `umap_only`
- `features` x `pca_then_umap`
- `pretrained_embeddings` x `pca_only`
- `pretrained_embeddings` x `umap_only`
- `pretrained_embeddings` x `pca_then_umap`

Inside each run, the expensive KMeans fits are parallelized with 16 workers.
This avoids running multiple heavy `joblib` pools on top of each other.

Aggregation is serial because it only reads JSON metadata and writes summary
tables.

## Outputs After A Full Run

Main summary files:

- `k_selection_summary.csv`
- `k_selection_summary.md`

Per-run output folders:

- `cluster_results/features/<reduction>/kmeans_gap_stability/`
- `cluster_results/pretrained_embeddings/<reduction>/kmeans_gap_stability/`

Each per-run folder contains:

- `selection_metrics.csv`: all candidate K values and their gap statistic,
  prediction strength, silhouette, Calinski-Harabasz, Davies-Bouldin, inertia,
  and cluster-size diagnostics.
- `k_selection_report.md`: short human-readable explanation for the selected K.
- `run_metadata.json`: machine-readable method parameters and final metrics.
- `cluster_assignments.csv`: final selected-K assignment for each song.
- `cluster_summary.csv`: final cluster sizes.
- `cluster_centers_reduced.npy`: selected KMeans centroids in reduced space.

## Decision Rule Recap

The implementation follows the shared ChatGPT link's decision hierarchy:

1. Primary K: gap statistic, smallest K satisfying the one-standard-error rule.
2. Stability check: prediction strength and bootstrap ARI.
3. Supporting CVIs: higher silhouette, higher Calinski-Harabasz, lower
   Davies-Bouldin.
4. If gap and stability disagree strongly, the report marks the run as needing
   interpretation review rather than treating the selected K as unquestionable.
